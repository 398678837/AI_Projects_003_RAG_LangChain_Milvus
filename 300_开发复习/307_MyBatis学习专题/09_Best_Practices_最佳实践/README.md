# MyBatis 最佳实践

## 1. 命名规范

### 1.1 Mapper 接口命名

- 使用 `XxxMapper` 命名，如 `UserMapper`
- 方法名使用动词+名词的形式，如 `getUserById`、`addUser`
- 方法名要清晰表达方法的功能

### 1.2 XML 映射文件命名

- 与 Mapper 接口同名，如 `UserMapper.xml`
- 放在与 Mapper 接口相同的目录下

### 1.3 SQL 语句 ID 命名

- 与 Mapper 接口方法名保持一致
- 使用小写字母和下划线，如 `get_user_by_id`

### 1.4 表名和列名命名

- 表名使用小写字母和下划线，如 `user_info`
- 列名使用小写字母和下划线，如 `user_name`

## 2. 目录结构

### 2.1 推荐目录结构

```
├── src/
│   ├── main/
│   │   ├── java/
│   │   │   └── com/
│   │   │       └── example/
│   │   │           ├── pojo/          # 实体类
│   │   │           ├── mapper/         # Mapper 接口
│   │   │           ├── service/        # 服务层
│   │   │           ├── controller/     # 控制器
│   │   │           └── utils/          # 工具类
│   │   └── resources/
│   │       ├── mybatis-config.xml      # MyBatis 配置文件
│   │       └── mapper/                 # XML 映射文件
│   └── test/
│       └── java/                       # 测试代码
```

### 2.2 注意事项

- Mapper 接口和 XML 映射文件放在不同的目录下，便于管理
- 实体类放在单独的包中，与数据库表对应
- 服务层负责业务逻辑，调用 Mapper 接口
- 控制器负责处理 HTTP 请求，调用服务层

## 3. SQL 编写

### 3.1 基本规范

- 使用小写字母编写 SQL 语句
- 使用缩进和换行，提高可读性
- 避免使用 SELECT *，只查询需要的字段
- 使用参数占位符 `#{}`，避免 SQL 注入

### 3.2 复杂 SQL 处理

- 对于复杂的 SQL 语句，使用 XML 映射文件
- 使用动态 SQL 标签处理条件查询
- 对于特别复杂的 SQL，可以考虑使用存储过程

### 3.3 性能优化

- 添加适当的索引
- 避免使用子查询，尽量使用连接查询
- 合理使用分页，避免一次性查询大量数据
- 使用 `limit` 限制查询结果数量

## 4. 参数传递

### 4.1 单个参数

- 直接使用 `#{参数名}`
- 对于基本类型，可以使用任意参数名

### 4.2 多个参数

- 使用 `@Param` 注解为参数命名
- 封装成对象传递
- 使用 Map 传递

### 4.3 集合参数

- 使用 `foreach` 标签处理数组和集合
- 注意 `collection` 属性的设置

## 5. 结果映射

### 5.1 基本映射

- 当列名与属性名一致时，使用 `resultType`
- 当列名与属性名不一致时，使用 `resultMap`

### 5.2 高级映射

- 使用 `<association>` 处理一对一关系
- 使用 `<collection>` 处理一对多关系
- 合理使用延迟加载，提高性能

### 5.3 类型转换

- 使用 `typeHandler` 处理特殊类型的转换
- 对于日期类型，指定 JDBC 类型

## 6. 缓存使用

### 6.1 一级缓存

- 默认开启，不需要额外配置
- 注意一级缓存的作用域是 SqlSession

### 6.2 二级缓存

- 对于不经常修改的数据，使用二级缓存
- 实体类需要实现 `Serializable` 接口
- 合理配置缓存的大小和淘汰策略

### 6.3 第三方缓存

- 对于复杂的应用场景，使用第三方缓存框架，如 Redis、EhCache 等
- 与 Spring 集成，使用 Spring 的缓存抽象

## 7. 事务管理

### 7.1 编程式事务

- 使用 `SqlSession` 的 `commit()` 和 `rollback()` 方法
- 适合简单的事务场景

### 7.2 声明式事务

- 与 Spring 集成，使用 `@Transactional` 注解
- 适合复杂的事务场景
- 配置事务的传播行为和隔离级别

### 7.3 注意事项

- 对于增删改操作，必须提交事务
- 事务的范围应该尽可能小，避免长时间占用数据库连接
- 避免在事务中执行耗时操作

## 8. 性能优化

### 8.1 SQL 优化

- 优化 SQL 语句结构
- 添加适当的索引
- 避免全表扫描
- 使用连接查询代替子查询

### 8.2 缓存优化

- 合理使用一级缓存和二级缓存
- 配置适当的缓存大小和淘汰策略
- 对于经常修改的数据，禁用缓存或设置较短的缓存时间

### 8.3 连接池优化

- 使用性能良好的连接池，如 HikariCP
- 配置适当的连接池大小
- 监控连接池的使用情况

### 8.4 批量操作

- 使用 `foreach` 标签进行批量插入、更新和删除
- 对于大量数据的批量操作，考虑使用批量处理 API

## 9. 代码组织

### 9.1 Mapper 接口

- 每个表对应一个 Mapper 接口
- 方法名要清晰表达方法的功能
- 避免在 Mapper 接口中添加业务逻辑

### 9.2 XML 映射文件

- 与 Mapper 接口保持一致
- 使用适当的缩进和换行，提高可读性
- 对于复杂的 SQL，添加注释

### 9.3 实体类

- 与数据库表对应
- 使用驼峰命名法
- 实现 `Serializable` 接口
- 添加适当的 getter 和 setter 方法

### 9.4 服务层

- 负责业务逻辑
- 调用 Mapper 接口
- 处理事务

## 10. 其他最佳实践

### 10.1 异常处理

- 捕获并处理 MyBatis 异常
- 向上层传递有意义的异常信息
- 记录异常日志

### 10.2 日志配置

- 配置适当的日志级别
- 对于 SQL 语句，使用 DEBUG 级别
- 对于生产环境，使用 INFO 级别

### 10.3 测试

- 为 Mapper 接口编写单元测试
- 测试各种场景，包括正常情况和异常情况
- 使用内存数据库进行测试

### 10.4 文档

- 为 Mapper 接口添加 JavaDoc 注释
- 为复杂的 SQL 语句添加注释
- 编写项目文档，说明项目结构和使用方法

### 10.5 版本控制

- 使用版本控制系统，如 Git
- 提交代码时添加有意义的提交信息
- 定期合并代码，避免冲突

## 11. 总结

MyBatis 是一个强大的持久层框架，通过遵循上述最佳实践，可以提高代码的质量和性能。在实际开发中，应该根据具体的需求和场景，选择合适的实现方式，以达到最佳的效果。
